# 测试指标记录

## 指标说明

- Unit Coverage：单元测试覆盖率，使用 `coverage.py` + `pytest-cov`。
- Unit Pass：单元测试通过数。
- Integration Pass：图级集成测试通过数。
- Intent Acc：意图识别准确率。
- Intent Macro F1：先分别计算推荐、预订、查询我的、普通问答四类意图的 F1，再做平均，避免样本多的类别掩盖样本少的类别。
- Slot Field F1：槽位字段级 F1，综合字段级 precision 和 recall。
- Slot Required Match：每条样本 expected 中的关键字段是否全部抽对；允许模型额外抽出非关键字段。
- Slot Exact Match：整条样本完全匹配；多抽字段也计为失败，是更严格的参考指标。
- SQL Safe：SQL 安全率，只允许只读 SELECT，并要求 LIMIT。
- SQL Exec：SQL 可执行率。
- E2E Success：端到端任务完成率。
- p95 Latency：压测 p95 延迟。

## 测试集评估总结

### 测试目的

本轮测试集评估用于验证 LLM 节点在真实自然语言输入下的语义能力，而不是替代单元测试。单元测试负责验证确定性代码路径、状态变化和图编译；测试集评估负责验证意图识别和槽位抽取是否能处理用户真实表达。

### 覆盖范围

当前测试集共 50 条：

- 意图识别：15 条，覆盖推荐房源、预订房源、查询我的、普通问答 4 类意图。
- 槽位抽取：15 条，覆盖预算范围、城市/区域、朝向、房型、推荐数量、设施要求、否定约束、通勤要求、多区域和信息缺失。
- SQL 约束：10 条，覆盖只读 SELECT、LIMIT、安全关键字、预算/区域/排序约束。当前尚未接入自动 runner。
- 端到端流程：10 条，覆盖推荐、补充信息、中断恢复、预订、查询历史和普通问答。当前尚未接入自动 runner。

本次改进来自两部分：

- 业务 prompt 改进：明确不根据地名推断城市、不把通勤地点当居住区域、不把整租/合租混入房型、多区域要完整保留。
- 评测口径改进：对“不要朝北，阳台”和“最好有阳台，不要朝北”这类顺序不同但语义一致的短语做等价匹配，同时把 `"None"`、`"无"`、`"无所谓"` 等非有效槽位值作为空值处理。

### 达标判断

当前不是所有样本 100% 通过，但按 v1/v2 阶段目标已经达标：

- 意图识别：Accuracy 100.00%，Macro F1 100.00%，超过 v2 目标。
- 槽位抽取：Field F1 92.93%，超过 v2 目标 88%。
- 关键字段完整命中率 86.67%，达到当前可接受水平；仍保留失败样本作为后续回归集。

允许少量失败的原因是 LLM 语义任务天然存在歧义，目标不是追求当前 15 条样本 100% 拟合，而是通过失败样本定位能力边界，并确保关键业务指标超过阈值。如果为了 100% 通过过度调整 prompt，反而可能对未见样本过拟合。

### 失败类型

当前剩余严格失败 5 条，其中 3 条关键字段已通过，只是多抽了额外字段：

- `slot_001`：把“海淀”额外抽成 city，同时 area 也正确为“海淀”。这是多抽字段问题，关键字段已通过。
- `slot_006`：额外抽取 `others=区域不限`。这是严格匹配失败，关键字段已通过。
- `slot_007`：额外抽取 `others=别太贵`。这是严格匹配失败，关键字段已通过。
- `slot_009`：`整租，不合租` 与 `整租，不要合租` 未被判等价。偏评测口径问题，也可通过同义词归一化改进。
- `slot_010`：`朝北的不要，优先朝南，最好有阳台` 抽成空对象。这是真失败，说明否定朝向 + 设施组合表达仍是模型弱点。

后续优化优先级：

1. 增加同义词归一化，例如“不合租”和“不要合租”判为等价。
2. 针对否定朝向和设施组合补充 prompt 示例。
3. 扩充 holdout 测试集，避免只围绕当前失败样本过拟合。
